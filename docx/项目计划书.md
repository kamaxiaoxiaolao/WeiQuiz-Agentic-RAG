# WeiQuiz 项目开发计划书

## 1. 项目定位

WeiQuiz 是一个面向企业知识库问答场景的 Agentic RAG 项目。项目目标不是只做一个简单 Demo，而是完成一个可以写进简历、可以现场演示、可以支撑面试问答的 LLM 应用工程项目。

最终定位：

> WeiQuiz：面向企业知识库的 Agentic RAG 问答系统，支持文档预处理、增量索引、混合检索、重排序、多轮对话、流式响应、来源追踪、评测闭环和 Agent 工具调用。

项目学习路线从数据预处理开始，逐步进入索引构建、检索优化、生成控制、多轮记忆、流式接口、评测体系、Agentic RAG 扩展和工程化部署。

## 2. 当前项目完整性评估

### 2.1 已完成的能力

当前项目已经具备一个 RAG 系统的基本闭环：

- 文档目录扫描。
- 文档 SHA256 增量 diff。
- PDF、Markdown、Text 基础解析。
- PDF 解析失败时回退到 `pypdf`。
- 文档清洗、标题修正、页眉页脚去重。
- chunk 切分。
- 稳定 `doc_id` 和 `chunk_id`。
- Milvus 向量库写入。
- LlamaIndex docstore 持久化。
- 向量检索。
- BM25 关键词检索。
- RRF 混合检索。
- DashScope rerank。
- FastAPI 后端服务。
- 单轮问答接口。
- 多轮聊天接口。
- Redis 会话记忆。
- SSE 流式输出。
- Streamlit UI。
- Vue 3 Demo UI。
- 基础 RAG 评测脚本。

### 2.2 当前不完整的地方

当前项目更接近“功能原型”，距离“简历级工程项目”还有这些差距：

- 数据预处理链路还没有形成标准化 pipeline。
- 文档格式支持还不完整，DOCX、HTML、复杂 PDF、表格、OCR 还需要增强。
- chunk 策略还比较基础，没有系统比较不同切分方式。
- 检索质量缺少量化评测。
- no-answer、低置信度拒答、澄清追问还没有实现。
- Agent 层还没有独立抽象。
- API 错误处理、日志、配置、测试、Docker Compose 还需要工程化。
- 文档和代码中存在中文乱码，需要清理。
- 旧的 Chroma/Ollama 实验代码和当前 Milvus 主链路还没有完全分离。

### 2.3 当前项目成熟度判断

可以这样判断：

- Demo 完整度：70%
- RAG 链路完整度：65%
- 数据预处理完整度：75%（主链路阶段性完成，深度清洗进入后续待办）
- 检索优化完整度：50%
- Agentic 能力完整度：20%
- 工程化完整度：35%
- 面试可讲性：已经具备基础，但需要通过优化和评测补强

## 3. 总体开发阶段

项目后续按 8 个阶段推进。

```text
Phase 1 数据预处理
  -> Phase 2 文档切分与索引
  -> Phase 3 检索优化
  -> Phase 4 生成与回答控制
  -> Phase 5 多轮记忆与流式体验
  -> Phase 6 评测闭环
  -> Phase 7 Agentic RAG 扩展
  -> Phase 8 工程化与简历包装
```

每个阶段都要同时完成三件事：

- 实现功能。
- 做至少一个优化点。
- 沉淀对应面试问题和回答。

## 4. Phase 1：数据预处理

### 4.0 阶段性结论

当前 Phase 1 可以标记为：**主链路阶段性完成，深度清洗作为后续待办**。

这里的“完成”不是说预处理已经做到生产极致，而是说它已经满足进入 Phase 2 的条件：文档能够被扫描、解析、清洗、转换为统一中间结构，并进入增量入库链路；同时具备基础 audit、ingestion report 和失败阶段记录，能够支持排查和面试讲解。

后续不再把所有精力继续压在预处理细节上。更细的 PDF 质量修复、OCR、复杂表格恢复、语义级去重、脱敏和解析质量评分，统一放入 Phase 1 后续增强待办，等检索、生成、评测主链路跑完整后再回来优化。

### 4.1 已完成能力

相关文件：

- `app/ingest/sync.py`
- `app/ingest/document_parser.py`
- `app/ingest/milvus_loader.py`
- `app/api.py`
- `frontend/weiquiz_vue.html`

当前已完成：

- 支持扫描 `data/docs` 下的 TXT / Markdown / PDF / DOCX / HTML 文档。
- 支持 SHA256 增量 diff，识别 added / updated / deleted。
- 支持稳定 `doc_id`，避免只用文件名导致同名冲突。
- PDF 优先使用 `unstructured` 解析，失败后 fallback 到 `pypdf`。
- DOCX / HTML / Markdown / TXT 通过 parser dispatch 进入统一解析链路。
- 将不同格式文档转换为统一 `Block` 中间结构。
- 做了基础文本清洗、标题修正、页眉页脚/噪声过滤。
- 支持按文件类型做差异化预处理策略。
- 生成 Markdown audit 文件，用于人工检查解析质量。
- 生成 ingestion report，记录成功、失败、失败阶段、文件类型统计、block/chunk 数量等。
- 支持上传文档 API、Redis job 状态、后台入库和前端上传页面。
- 修复 DashScope embedding batch size 限制，避免批量入库 400 错误。

### 4.2 当前不足

当前预处理还有不足，但不再阻塞 Phase 2：

- PDF 转 Markdown 的质量仍不稳定，公式、特殊字体、复杂版式可能出现乱码或碎片。
- 对扫描版 PDF 还没有完整 OCR 链路。
- 表格结构只做了基础 Markdown 化，还没有复杂跨页表格恢复。
- 文档质量统计还比较基础，没有形成完整的数据质量评分。
- 敏感信息脱敏、重复文档检测、近重复内容识别还未实现。
- 中文乱码问题在部分历史文档和 audit 中仍可能出现，需要后续专项清理。

### 4.3 后续待办：预处理增强项

这些内容后续可以继续优化，但不作为当前主线阻塞项：

- PDF 解析质量增强：布局识别、公式噪声识别、页眉页脚更稳健去重。
- OCR 支持：针对扫描版 PDF 接入 OCR，并在 report 中标记 OCR 使用情况。
- 表格增强：识别表头、跨页表格、表格说明，转成更适合检索的 Markdown。
- 文档质量评分：统计空 block、超短 block、乱码比例、重复率、表格占比。
- 文档去重：支持 hash、SimHash、MinHash 等不同粒度去重。
- 敏感信息脱敏：手机号、邮箱、身份证、API key 等入库前处理。
- 数据质量评测：把预处理质量和后续 Recall@k、MRR、答案准确性关联起来。

### 4.4 面试表达重点

面试中不要说“预处理已经完全做好了”，而要这样表达：

> 我们当前把预处理做成了一条可追踪的 ingestion pipeline。不同格式文档会先经过 parser dispatch，统一转换成 Block，再做基础清洗、audit、metadata 补充和增量入库。这样做的好处是后续 chunk、检索和溯源都有统一的数据基础。当前版本已经能支持 TXT、Markdown、PDF、DOCX、HTML 的入库和排错，但 PDF 复杂版式、OCR、表格恢复、文档质量评分这些属于更深层的数据质量优化，我把它们作为后续增强项，不阻塞当前进入 chunking、检索和评测主链路。

### 4.5 下一阶段入口

Phase 1 接下来只保留必要维护，不继续深挖细节。

主线进入 Phase 2：**文档切分与索引构建**。

Phase 2 的首个重点问题：

- 当前 chunking 是否只是固定 token 切分？
- 如何避免标题、表格、条款、知识点被切散？
- 是否需要 parent-child chunk、section_path、chunk 质量统计？
- 如何把 chunking 策略讲成一个能过面试的工程方案？

## 5. Phase 2：文档切分与索引构建

### 5.0 阶段定位

本阶段目标不是简单回答“chunk_size 设多少”，而是把文档切分从默认窗口切分升级为可解释、可评估、可继续扩展的 chunking 方案。

当前项目的准确实现定位是：**结构化预聚合 + LlamaIndex SentenceSplitter**。

也就是说，当前不是完整的语义切分、层级切分或 parent-child chunk。最终生成 chunk 的仍然是 LlamaIndex `SentenceSplitter`，但在调用 splitter 之前，项目已经把解析出来的碎 Block 聚合成更完整的 section，并合并过短 section，避免大量低信息密度 chunk 进入向量库。

### 5.1 当前已完成能力

相关文件：

- `app/ingest/document_parser.py`
- `app/ingest/milvus_loader.py`
- `app/eval/retrieval_eval.py`
- `data/eval/retrieval_questions.jsonl`

当前链路：

```text
Block
  -> clean / fix_titles
  -> section-level Document
  -> short section merge
  -> LlamaIndex SentenceSplitter
  -> TextNode chunk
  -> Milvus / Docstore
```

已完成优化：

- 增加 `chunk_stats`，统计 chunk 数量、长度分布、短 chunk、长 chunk、按文档和文件类型聚合结果。
- 将“每个 Block 直接进入 splitter”改为“多个 Block 先聚合成 section-level Document”。
- 增加短 section 合并策略，避免孤立标题、短句、公式碎片、列表残片单独成为 chunk。
- 精简入库 metadata，只保留检索、展示和基础溯源所需字段。
- 增加 `section_md` audit，用于人工检查 section 合并结果。
- 增加轻量检索评估脚本，用 Hit@K / MRR / latency 检查正确上下文是否被召回。
- 完成一次 badcase 分析，发现首次低分主要来自评测集标注问题，而不是直接证明检索策略失败。

### 5.2 当前效果

当前优化前后对比：

```text
初始 Block 直接切分：
total_chunks       548
avg_chunk_length   22.37
p50                11
short_chunk_count  479

section 聚合后：
total_chunks       156 / 144
avg_chunk_length   87 - 94
p50                41 - 49
short_chunk_count  79

短 section 合并后：
total_chunks       53
avg_chunk_length   259.87
p50                201
short_chunk_count  0
```

结论：当前结构化预聚合是有效的。它解决的是 splitter 输入过碎的问题，但还没有完成 parent-child、semantic chunking 或 hierarchical chunking。

### 5.3 主流策略分类

Phase 2 中需要理解和对比的主流 chunking 策略包括：

1. 固定窗口切分：按 token / 字符长度切分，配合 overlap，适合作 baseline。
2. 句子 / 递归切分：优先按段落、换行、句号等边界切分，例如 LlamaIndex SentenceSplitter、LangChain RecursiveCharacterTextSplitter。
3. 结构化切分：按标题、章节、段落、表格、FAQ、条款等文档结构切分。
4. 语义切分：根据 embedding 相似度或主题变化判断边界。
5. 层级切分：生成 root / parent / child 多层 chunk。
6. 父子块切分：child chunk 用于精确召回，parent chunk 用于补充上下文。
7. 业务结构切分：例如法律按条款，题库按题目 / 解析 / 知识点，表格按行或表摘要处理。

当前项目已经完成的是第 2 类的基础能力和第 3 类的预聚合增强：**结构化预聚合 + 句子窗口切分**。

### 5.4 后续优化计划

P0：当前阶段必须完成

- 保留当前 `section_merge + SentenceSplitter` 作为默认策略。
- 用 `chunk_stats` 和 `retrieval_eval` 形成策略评估闭环。
- 明确当前实现边界，避免把当前方案误称为完整结构化切分或语义切分。

P1：建议实现一个主流增强

- parent-child chunk：child 负责精准召回，parent 负责补充上下文。
- parent chunk 可以复用当前 section-level Document。
- child chunk 可以继续使用 SentenceSplitter 生成。
- 检索命中 child 后，根据 parent_id 找回 parent context 给 LLM。

P2：记录到计划，不在当前小节展开

- semantic chunking。
- hierarchical multi-level chunk。
- 表格专用 chunk。
- 业务结构 chunk。
- 使用 Ragas / LlamaIndex 官方评估能力做完整评测闭环。

### 5.5 面试表达重点

不要说：

> 我已经实现了完整结构化切分 / 语义切分。

应该说：

> 我当前复用了 LlamaIndex SentenceSplitter 作为基础切分器，但在 splitter 之前增加了结构化预聚合。因为 PDF 解析出来的 Block 很碎，直接切会产生大量短 chunk。我先把 Block 按标题和长度聚合成 section，再合并短 section，最后交给 SentenceSplitter 切分。这个方案属于结构感知的切分优化，不是完整 semantic chunking 或 parent-child chunk。后续我会扩展 parent-child chunk，用 child 做召回，parent 补上下文。

高频问题：chunk size 和 overlap 怎么定？

回答要点：

- chunk size 是语义完整性、召回精度和成本之间的平衡。
- 太小会导致语义碎片化，标题、定义、表格、知识点容易被切散。
- 太大会混入无关内容，增加上下文噪声、rerank 成本和 LLM 成本。
- overlap 用于缓解切分边界问题，但过大会导致重复 chunk、向量库膨胀和召回重复。
- 不能只凭经验选参数，要结合 chunk_stats、Hit@K、MRR、引用准确率和 latency 做对比。

高频问题：怎么证明 chunking 优化有效？

回答要点：

- 第一层看 chunk_stats，排除明显异常策略。
- 第二层看 retrieval_eval，用 Hit@K 和 MRR 判断正确上下文是否被召回、排名是否靠前。
- 第三层后续看生成评估，例如 faithfulness、answer relevancy、context precision、context recall。
- 指标低时先做 badcase analysis，判断是评测集标注、解析、切分还是检索策略问题。

## 6. Phase 3：检索优化

### 6.0 企业级推进原则

本阶段重点从“能检索”升级为“可评测、可调参、可解释的检索系统”。

- P0：检索模式对比。实现 vector only、BM25 only、hybrid、hybrid + rerank 的可切换对比。
- P0：BM25 增量维护。后续补 increment_add / increment_remove，避免文档更新后 BM25 统计和向量库不一致。
- P1：metadata filter。支持按文档、文件类型、章节、时间等条件过滤。
- P1：rerank 策略。比较 rerank 前后召回结果，记录 score、rank、source。
- P2：query rewrite、multi-query、router retriever 等优化必须通过评测证明收益后再推进。

本阶段面试目标：能回答“为什么 hybrid retrieval、BM25 和向量检索各解决什么问题、怎么评估检索优化有效”。

### 6.1 当前实现

当前检索链路：

```text
Milvus vector retriever
  + BM25 retriever
  -> QueryFusionRetriever RRF
  -> DashScopeRerank
```

### 6.2 优化计划

短期优化：

- 增加检索模式开关：vector only、BM25 only、hybrid、hybrid + rerank。
- 在 API 返回每个来源的 retriever 类型、score、rank。
- UI 展示来源分数和 chunk metadata。

中期优化：

- 增加 query rewrite。
- 增加 metadata filter。
- 增加多查询扩展。
- 增加 no-answer 阈值。

高级优化：

- 针对不同问题类型选择不同检索策略。
- 增加 router retriever。
- 对召回和精排分别做评测。

### 6.3 面试高频问题

问题：为什么 hybrid retrieval 比单纯向量检索好？

回答要点：

- 向量检索擅长语义匹配。
- BM25 擅长关键词和专有名词。
- 企业文档中经常有政策编号、API 名称、产品名，BM25 很有价值。
- hybrid 可以提高召回稳定性。

问题：RRF 的作用是什么？

回答要点：

- RRF 基于排名融合多个检索器结果。
- 不直接比较不同检索器的原始分数。
- 适合融合向量检索和 BM25。

问题：rerank 放在哪里？为什么？

回答要点：

- rerank 放在召回之后。
- 先用便宜检索器扩大候选集，再用较贵的 rerank 做精排。
- 这样兼顾召回、精度和成本。

## 7. Phase 4：生成与回答控制

### 7.0 企业级推进原则

本阶段重点从“能生成答案”升级为“答案可控、可拒答、可溯源”。

- P0：引用控制。回答必须带来源，来源要能回到 chunk / document metadata。
- P0：no-answer / 低置信度拒答。上下文不足时不强答。
- P1：prompt 模板结构化。区分系统指令、用户问题、检索上下文、输出格式。
- P1：答案校验。检查答案是否基于上下文，是否出现无来源断言。
- P2：风格优化、小提示词优化必须服务于准确性和可控性，不作为主线。

本阶段面试目标：能回答“怎么降低幻觉、怎么做拒答、怎么保证答案来自检索上下文”。

### 7.1 当前实现

当前通过 LlamaIndex query engine 和 chat engine 生成回答，并返回 source nodes。

### 7.2 优化计划

短期优化：

- 增加统一 prompt，要求基于上下文回答。
- 当上下文不足时明确回答“知识库中没有找到依据”。
- 返回引用来源。

中期优化：

- 增加 answer verification。
- 增加 groundedness 检查。
- 增加回答格式控制，例如摘要、列表、步骤。

高级优化：

- 支持不同任务类型：问答、总结、对比、抽取、生成操作建议。
- 让 Agent 根据问题选择生成策略。

### 7.3 面试高频问题

问题：如何减少 RAG 幻觉？

回答要点：

- 提升检索质量。
- 使用 rerank。
- 加 no-answer 机制。
- prompt 限制必须基于上下文。
- 展示来源。
- 使用 faithfulness 评测。
- 加 answer verification。

## 8. Phase 5：多轮记忆与流式体验

### 8.0 企业级推进原则

本阶段重点从“能多轮聊天”升级为“会话状态可管理、成本可控、体验稳定”。

- P0：会话记忆策略。区分短期上下文、长期偏好、任务状态。
- P0：历史裁剪。按轮次和 token budget 控制上下文长度。
- P1：Redis 可靠性。TTL、降级、异常处理、session 管理。
- P1：SSE 流式稳定性。处理中断、超时、错误消息和前端体验。
- P2：长期记忆、用户画像等高级能力后置，先保证基础多轮链路可靠。

本阶段面试目标：能回答“多轮记忆怎么存、怎么避免上下文爆炸、Redis 挂了怎么办、流式输出怎么处理异常”。

### 8.1 当前实现

当前项目支持：

- Redis 保存 `ChatMemoryBuffer`。
- session 列表。
- session 删除。
- session 消息读取。
- `/chat/stream` SSE 流式输出。

### 8.2 优化计划

短期优化：

- 统一 SSE 事件格式。
- 增加错误事件。

## 13. 主流先进技术扩展计划

这个阶段的目标不是盲目堆技术，而是围绕当前 Agent / RAG / LLM 应用开发岗位常见的先进方向，选择适合 WeiQuiz 的能力逐步扩展。每个扩展都要能回答三个问题：

- 它解决什么问题？
- 为什么当前项目需要它？
- 面试时如何讲清楚它的设计取舍？

### 13.1 Graph RAG / 知识图谱 RAG

Graph RAG 的核心思想是：普通 RAG 主要检索文本 chunk，而知识图谱 RAG 会额外抽取实体、关系和路径，用结构化知识增强问答。

适合场景：

- 多跳推理。
- 关系查询。
- 依赖分析。
- 组织架构问答。
- 业务流程问答。
- 需要强可解释性的企业知识问答。

扩展计划：

- 从 chunk 中抽取实体关系三元组：`(subject, relation, object)`。
- 每条关系保留 `doc_id`、`chunk_id`、`source_path`、`confidence`。
- 先使用 JSON 或 SQLite 保存图谱数据。
- 后续增加 `app/graph/` 模块，包括 `schema.py`、`extractor.py`、`store.py`、`retriever.py`、`prompts.py`。
- 增加 `GraphSearchTool`，让 Agent 根据问题类型选择文本检索或图谱检索。
- 高级阶段可接入 Neo4j、NebulaGraph 或 NetworkX，支持多跳路径查询和图谱可视化。

面试问题：Graph RAG 和普通 RAG 有什么区别？

回答要点：

- 普通 RAG 面向文本片段检索。
- Graph RAG 会抽取实体和关系，构建结构化知识。
- Graph RAG 更适合多跳推理、关系查询和依赖分析。
- 最好的方式通常不是替代文本 RAG，而是文本检索和图谱检索融合。

### 13.2 多模态 RAG

多模态 RAG 关注的不只是文本，还包括 PDF 图片、表格、截图、流程图、PPT、扫描件等信息。

适合场景：

- 扫描版 PDF。
- 产品手册中的图片说明。
- 架构图和流程图问答。
- 表格密集型文档。
- PPT 知识库。

扩展计划：

- 在 PDF 解析中识别图片、表格和文本块。
- 对表格转 Markdown。
- 对图片保存 `image_path` 和 `page_no`。
- 对图片或页面截图生成 caption。
- 将 caption 作为可检索文本入库。
- 高级阶段支持多模态 embedding、图文联合检索和页面级证据返回。

面试问题：多模态 RAG 和文本 RAG 的区别是什么？

回答要点：

- 文本 RAG 只处理文本 chunk。
- 多模态 RAG 需要处理图片、表格、版面结构和 OCR。
- 难点在于不同模态如何表示、如何检索、如何对齐来源。

### 13.3 长期记忆与用户画像

Agent 系统中的 memory 不只是聊天历史，还可以分为短期记忆、长期记忆、任务状态和用户偏好。

扩展计划：

- 将 Redis 中的会话记忆分为短期上下文和会话摘要。
- 长对话时自动总结历史消息。
- 增加用户偏好记忆。
- 增加任务状态记忆。
- 支持跨会话恢复上下文。
- 高级阶段使用向量库保存长期记忆，并设计记忆写入策略，避免无意义内容进入长期记忆。

面试问题：Agent 记忆怎么设计？

回答要点：

- 短期记忆用于当前对话。
- 长期记忆用于跨会话信息。
- 需要摘要压缩、重要性判断和过期策略。
- 不能把所有聊天内容无脑写入长期记忆，否则会引入噪声和隐私风险。

### 13.4 Agent 工作流与多 Agent 协作

Agent 不只是 tool calling，还包括任务规划、工具选择、执行反馈、反思和多 Agent 协作。

扩展计划：

- 增加显式 Agent 执行轨迹。
- 增加工具调用日志。
- 增加 query router。
- 增加 planner。
- 增加 reflection。
- 增加 answer verifier。
- 高级阶段可设计多个角色：Retriever Agent、Verifier Agent、Writer Agent。

面试问题：什么时候需要多 Agent？

回答要点：

- 简单问答不需要多 Agent。
- 当任务需要规划、检索、分析、验证、写作等多个阶段时，多 Agent 可以拆分职责。
- 多 Agent 的问题是成本、延迟和调试复杂度更高。

### 13.5 Context Engineering / 上下文工程

上下文工程是 LLM 应用的核心能力之一，不只是写 prompt，还包括检索内容选择、压缩、排序、格式化和引用控制。

扩展计划：

- 统一上下文模板。
- 控制 chunk 数量和总 token。
- 按相关性和来源组织上下文。
- 增加 context compression。
- 增加 lost-in-the-middle 缓解策略。
- 增加引用编号。
- 高级阶段根据问题类型动态构造上下文。

面试问题：上下文太长怎么办？

回答要点：

- 先提升检索精度。
- 使用 rerank 选出更少但更相关的 chunk。
- 做 context compression。
- 对长上下文进行摘要。
- 控制 prompt 结构，减少无关信息。

### 13.6 评测、Tracing 与可观测性

先进 LLM 应用不能只看最终答案，还要能追踪检索、prompt、模型输出、延迟、成本和错误。

扩展计划：

- 生成本地 Markdown 评测报告。
- 记录检索结果、rerank 结果、最终 prompt。
- 接入 Langfuse 或 Phoenix。
- 跟踪每次请求的 latency、TTFT、token 使用量。
- 支持失败案例回放。
- 建立回归评测集，每次优化后自动对比指标。

面试问题：LLM 应用如何做可观测性？

回答要点：

- 需要记录输入、检索上下文、prompt、模型输出、token、延迟、错误。
- RAG 还要记录召回结果和来源。
- 评测和 tracing 可以帮助定位到底是检索问题、prompt 问题还是模型问题。

### 13.7 模型与部署优化

真实项目中不仅要效果好，还要考虑成本、延迟、稳定性和部署方式。

扩展计划：

- 区分在线大模型和本地模型配置。
- 支持 DashScope、Ollama 两类模型入口。
- 增加 embedding cache。
- 增加 response cache。
- 增加 rate limit。
- 支持模型路由：简单问题用小模型，复杂问题用大模型。
- 对比不同模型在效果和成本上的差异。

面试问题：如何降低 LLM 应用成本？

回答要点：

- 缓存 embedding、检索结果和部分回答。
- 控制上下文长度。
- 简单任务走小模型。
- 复杂任务才调用大模型或 rerank。
- 用评测保证降本不明显损害效果。

### 13.8 扩展阶段的推荐执行顺序

建议不要一口气做完所有先进技术，而是按和当前项目的贴合度推进：

1. 先做 Graph RAG 原型，因为它和当前文档 chunk、metadata、Agent tool 最容易衔接。
2. 再做 Context Engineering，因为它能直接提升当前 RAG 答案质量。
3. 然后做 Tracing 和评测报告，用数据展示优化效果。
4. 再做长期记忆，把 Redis 会话升级成更完整的 Agent Memory。
5. 最后做多模态 RAG 和多 Agent 协作，这两个方向更高级，但工程复杂度也更高。

## 15. 文档入库链路高级优化参考

这一节用于吸收更工程化的文档入库设计。它不是当前必须一次性完成的功能，而是后续优化时的重要参考。

参考链路：

```text
上传文档
  -> 判断同名或同 doc_id 文档是否已存在
  -> 删除旧文档相关索引状态
  -> 文档解析与清洗
  -> 层级 chunk 切分
  -> 写入 parent chunk store
  -> 更新 BM25 统计
  -> 生成 dense / sparse 向量
  -> 写入 Milvus
  -> 后续检索直接使用新文档参与召回
```

### 15.1 索引一致性

当前项目已经有“更新文档时先删除旧内容，再写入新内容”的基础意识。但后续需要进一步扩展为“删除所有相关索引状态”。

一个文档可能同时存在于：

- LlamaIndex docstore。
- Milvus 向量库。
- BM25 关键词索引。
- parent chunk store。
- Graph RAG 的实体和关系存储。
- ingest_state / ingestion report。

因此，文档更新时不能只删除 Milvus 中的向量。更稳的做法是以 `doc_id` 为中心，清理所有相关状态，再重新解析和写入。

面试表达：

> 文档更新时，只删除向量库是不够的。RAG 系统中同一篇文档可能同时存在于 docstore、向量库、BM25 统计、父块缓存和图谱关系中。如果只删一处，就会出现索引不一致。我的设计会以 `doc_id` 为中心维护文档关联状态，更新时先删除旧 chunk 及相关索引，再写入新版本。

### 15.2 层级 chunk 设计

当前项目主要是 flat chunk：

```text
doc_id::chunk::0
doc_id::chunk::1
doc_id::chunk::2
```

后续可以扩展为层级 chunk：

```text
root chunk    文档级或大章节级
parent chunk  章节级或段落组级
child chunk   最终用于检索的小块
```

建议预留 chunk metadata：

```text
chunk_id
parent_chunk_id
root_chunk_id
chunk_level
```

第一阶段可以先这样设置：

```text
chunk_id = doc_id::chunk::0
parent_chunk_id = null
root_chunk_id = doc_id
chunk_level = 0
```

后续做 parent-child retrieval 时再真正启用层级结构。

面试表达：

> 小 chunk 检索更精确，但上下文可能不完整；父 chunk 能提供更完整的语义上下文。因此我会设计层级 chunk，检索时命中 child chunk，生成时可以补充 parent chunk，兼顾召回精度和上下文完整性。

### 15.3 Parent Chunk Store

Parent Chunk Store 用于保存父级 chunk 或章节级上下文。

作用：

- child chunk 用于精准召回。
- parent chunk 用于补充回答上下文。
- root chunk 用于文档级追踪。
- Graph RAG 可以把实体关系绑定到 parent 或 section。

后续可以实现：

```text
命中 child chunk
  -> 根据 parent_chunk_id 找到 parent chunk
  -> 将 parent chunk 作为上下文补充给 LLM
```

这适合长文档、制度文档、技术手册等场景。

### 15.4 BM25 增量统计

当前项目的 BM25 是基于 docstore 中所有节点构建的：

```text
BM25Retriever.from_defaults(nodes=all_nodes)
```

小规模项目够用，但文档变多后会有问题：

- 服务启动时构建 BM25 较慢。
- 每次更新文档后不容易只更新局部统计。
- BM25 状态和向量库可能出现不一致。

后续可以设计 BM25 增量维护：

```text
文档删除或更新
  -> 对旧 chunk 执行 increment_remove

文档新增或更新
  -> 对新 chunk 执行 increment_add
  -> 更新 bm25_state.json 或 SQLite 状态
```

这可以作为 Phase 3 检索优化中的高级任务。

面试表达：

> 当前项目里 BM25 是启动时基于 docstore 重建的，适合原型阶段。后续如果文档规模变大，我会维护 BM25 的增量统计状态，文档更新时先 remove 旧 chunk 的词频统计，再 add 新 chunk，保证 BM25 与向量索引一致。

### 15.5 Dense + Sparse Hybrid Search

当前项目使用的是：

```text
Milvus dense vector
  + LlamaIndex BM25
  + RRF fusion
```

这是主流方案之一。

另一种路线是：

```text
Milvus dense vector
  + sparse vector
  -> Milvus hybrid search
```

两种路线都可以用于 hybrid retrieval：

- Dense vector 擅长语义相似。
- Sparse vector / BM25 擅长关键词、编号、专有名词。

后续可以在 Phase 3 中做对比实验：

```text
方案 A：Dense + BM25 + RRF
方案 B：Milvus Dense + Sparse Hybrid Search
```

评测指标：

- hit@k
- MRR
- faithfulness
- answer relevancy
- latency

面试表达：

> Hybrid retrieval 有多种实现方式。当前项目采用 Milvus dense vector 加 LlamaIndex BM25，再通过 RRF 融合。后续也可以尝试 Milvus 原生 dense+sparse hybrid search，并用评测集比较两种方案在召回质量和延迟上的差异。

### 15.6 对当前阶段的影响

当前阶段不直接实现完整层级 chunk、BM25 增量或 dense+sparse hybrid search。当前只做两件事：

1. 在 metadata 设计中预留可扩展字段：

```text
parent_chunk_id
root_chunk_id
chunk_level
```

2. 在面试表达中说明后续演进路线：

```text
flat chunk
  -> parent-child chunk
  -> parent chunk store
  -> BM25 增量统计
  -> dense+sparse hybrid search
  -> Graph RAG 来源追踪
```

这样既不会让当前实现过早复杂化，也能体现项目具备面向主流 RAG 工程方向的扩展设计。

### 17.2 Phase 2：Chunking 面试逻辑

典型面试问题：

> 在你项目的 RAG 实现中，文本分块 Chunking 策略是如何设计的？遇到过哪些问题，如何调整的？

后续 Phase 2 开发必须围绕这个问题补齐证据。

#### 选型为什么

第一版可以使用 LlamaIndex `SentenceSplitter` 作为 baseline，因为它实现简单，能快速打通文档切分、向量化和检索链路。

但面试中不能只停留在“用了 512 token 切分”。需要说明：固定 token 切分只是 baseline，后续会根据文档结构和评测结果调整。

#### 遇到什么问题

需要在项目中能观察或复现这些问题：

- 标题和正文被切开。
- 表格被切散。
- 条款、FAQ、题目解析等完整业务语义被切散。
- chunk 太短，语义不足，召回后无法支撑回答。
- chunk 太长，噪声变多，rerank 和生成成本上升。
- overlap 太大，导致索引重复、存储成本和检索噪声增加。

#### 怎么解决

Phase 2 需要逐步实现或预留：

- `block_id`，建立 `doc_id -> block_id -> chunk_id` 链路。
- `section_path`，保留标题层级和章节路径。
- chunk 统计报告，记录 chunk 总数、平均长度、超短/超长比例。
- 对比不同 `chunk_size` 和 `chunk_overlap`。
- 对强结构文档使用结构化切分，例如标题、章节、条款、FAQ、表格。
- 预留 parent-child chunk 或 hierarchical chunk 扩展。

#### 怎么评测

Chunking 优化不能只凭主观感觉，需要用指标或统计证明：

- chunk 总数
- 平均 chunk 长度
- P50 / P90 / P95 chunk 长度
- 超短 chunk 比例
- 超长 chunk 比例
- Recall@k / hit@k
- MRR
- faithfulness
- answer relevancy
- latency
- token cost

至少要对比：

```text
baseline：固定 token 切分
optimized：结构化切分 / 带 section_path 的切分
```

#### 业务价值

Chunking 决定了 RAG 检索召回的最小语义单元。切分太碎，模型拿不到完整依据；切分太粗，召回内容噪声高。好的 chunk 策略能提升检索命中质量、降低幻觉、提高答案可溯源性，并减少无效上下文带来的成本。

#### 面试回答目标

> 我第一版使用 LlamaIndex SentenceSplitter 做固定 token 切分，主要是为了快速建立 baseline。后面我发现固定切分容易把标题和正文、表格、条款或完整业务语义切散，所以在预处理阶段保留了 Block 结构和 metadata，并在切分阶段补充 block_id、section_path 和 chunk 统计报告。后续通过对比不同 chunk_size、overlap 和结构化切分策略，用 Recall@k、MRR、faithfulness、latency 等指标评估效果，而不是只凭经验调整参数。

## 19. 上传触发型增量更新计划

当前项目已经实现目录扫描型增量索引：扫描 `docs_dir`，计算 sha256，对比 `ingest_state`，得到 added / updated / deleted，然后对变化文档执行入库或删除。

这适合作为本地开发、批量同步和定时任务的 baseline。但如果项目要做成更像真实产品的知识库系统，还需要支持上传触发型增量更新。

### 19.1 为什么需要上传触发型增量

产品型知识库系统中，用户通常不是手动把文件放进目录，而是通过前端或后台管理系统上传文档：

```text
POST /documents/upload
  -> 保存原始文件
  -> 生成或读取 doc_id
  -> 判断是否已存在旧版本
  -> 删除旧 chunk
  -> 解析、清洗、切分
  -> 写入向量库 / BM25 / docstore
  -> 生成 ingestion report
```

因此，上传触发型增量更接近真实业务系统。

### 19.2 与当前目录扫描型增量的关系

两种方式不应该写成两套完全独立逻辑。

推荐架构是抽象统一 ingestion service：

```text
目录扫描 diff
        -> ingestion service

前端上传文件
        -> ingestion service
```

统一的 ingestion service 负责：

- 删除旧版本 chunk。
- 文档解析。
- 文本清洗。
- metadata 标准化。
- chunk 切分。
- 写入 Milvus。
- 更新 BM25。
- 写入 docstore。
- 生成 ingestion report。

这样可以避免“扫描同步”和“上传同步”两条链路逻辑分裂。

### 19.3 后续需要补的能力

上传触发型增量需要补充：

- `POST /documents/upload` 接口。
- 文档记录表或 metadata store。
- 文件版本管理。
- 单文档 upsert。
- 同名文件 / 同 doc_id 文件覆盖策略。
- 上传失败回滚策略。
- 上传任务状态查询。
- 前端上传 UI。

### 19.4 与其他阶段的关系

这个能力不在当前 Phase 1 立刻实现，按阶段拆分：

- Phase 1：继续完善 ingestion pipeline、metadata、ingestion report。
- Phase 2：完善 chunk 层级和 parent-child chunk 能力。
- Phase 3：补 BM25 increment_add / increment_remove。
- Phase 8：实现上传接口、文档管理、版本管理和前端上传 UI。

### 19.5 面试表达

> 当前项目先实现目录扫描型增量索引，通过 sha256 和状态文件判断新增、修改、删除，适合批量同步知识库。产品化场景下，我会进一步支持上传触发型增量更新：用户上传文档后，对单个 doc_id 做 upsert，如果旧版本存在，先删除旧 chunk，再重新解析、切分并写入向量库和 BM25。底层复用同一套 ingestion pipeline，避免扫描同步和上传同步两套逻辑分裂。

- 上传链路已可用，但还缺文档版本表、上传回滚、权限校验、文件去重和异步队列。
- 核心预处理函数缺少 pytest 单元测试。

对齐优先级：

- P0：保留现有预处理主链路，不继续深挖零散清洗细节。
- P1：补充质量门禁：乱码比例、空块比例、短块比例、解析失败率。
- P1：给 `sync.py`、`document_parser.py`、`milvus_loader.py` 增加核心单元测试。
- P2：OCR、复杂表格恢复、敏感信息脱敏、文档去重。

### 20.3 Phase 2 文档切分与索引构建

当前实现：

- 当前准确策略是“结构化预聚合 + LlamaIndex SentenceSplitter”。
- `Block` 会先合并成 section-level `Document`，避免碎 block 直接进入 splitter。
- 已实现短 section 合并，chunk_stats 从大量超短 chunk 降到更合理分布。
- metadata 已精简，避免 LlamaIndex metadata 长度超过 chunk_size。
- 已生成 `section_md` audit，用于人工检查合并后的 section 质量。
- 已新增 PostgreSQL parent section store 基础模块 `app/storage/parent_store.py`，并在入库链路中预留 parent section 写入。

企业级差距：

- 当前还不是完整 parent-child retrieval：PostgreSQL 已作为 parent store 基础，但 child 命中后回查 parent 并组装上下文的检索链路还未闭环。
- 表格、代码块、公式块仍可能被 SentenceSplitter 普通切分，需要类型感知切分。
- 当前只有一套实际策略，缺少 baseline 对比：固定窗口、SentenceSplitter、结构化预聚合、parent-child 的系统对比。
- PostgreSQL 连接和库初始化还需要本地环境验证；`uv.lock` 未同步 psycopg 依赖。
- Chunk 质量统计已有，但还没有门禁策略，例如短 chunk 超过阈值时拒绝入库。

对齐优先级：

- P0：完成 PostgreSQL parent store 环境验证，确保 parent section 能写入和查询。
- P0：完成 child chunk 命中后回查 parent section 的 parent-child retriever。
- P0：实现类型感知切分：表格/代码块/公式块不走普通 SentenceSplitter。
- P1：做 chunk 策略对比实验，并输出 Hit@K、MRR、latency、chunk_stats 对比报告。
- P1：把 dense_top_k、sparse_top_k、fusion_top_k、rerank_top_n、final_top_k 参数拆开。

### 20.4 Phase 3 检索优化

当前实现：

- `app/rag_milvus.py` 已实现向量检索 + BM25 + QueryFusionRetriever RRF 融合。
- 已接入 DashScope rerank。
- `app/eval/retrieval_eval.py` 支持 vector / hybrid 两种模式的基础检索评测。

企业级差距：

- BM25 目前依赖本地 docstore 节点重建，缺少独立持久化和增量维护能力。
- 检索参数写死较多，缺少配置化和实验化。
- 没有查询重写、HyDE、query expansion、意图路由。
- 没有对低质量检索结果做 fallback，例如扩大 top_k、改写 query、拒答或澄清追问。
- 缺少按业务问题类型的 badcase 分类。

对齐优先级：

- P0：先把 top_k 参数拆开并配置化。
- P1：建立 vector / BM25 / hybrid / hybrid+rerank 的评测对比。
- P1：补 query rewrite 或 HyDE 作为检索增强实验。
- P2：BM25 工程化持久化，生产级可考虑 OpenSearch / Elasticsearch / PostgreSQL full-text / Milvus sparse vector。

### 20.5 Phase 4 生成与回答控制

当前实现：

- 已有 `/query` 单轮问答接口。
- 已通过 LlamaIndex `RetrieverQueryEngine` 使用检索结果生成回答。
- 返回 source_nodes，包含文本、score、source_path、chunk_id。

企业级差距：

- 没有统一 prompt 模板和回答约束。
- 没有低置信度拒答、no-answer 检测、引用格式控制。
- 没有基于 source 的答案校验，容易出现“检索到了但回答没引用依据”的问题。
- 没有结构化输出，难以支撑工具调用、前端可控展示和自动评测。

对齐优先级：

- P0：增加回答约束 prompt：必须基于上下文回答，证据不足时拒答。
- P1：增加 citation/source 格式控制。
- P1：增加 no-answer 阈值和澄清追问策略。
- P2：用 Pydantic 定义结构化回答对象。

### 20.6 Phase 5 多轮记忆与流式体验

当前实现：

企业级差距：

- SSE 没有完整错误事件、取消任务、客户端断连处理。

对齐优先级：

- P1：增加记忆裁剪和摘要策略。
- P1：增加 stream error event 和前端异常展示。
- P2：长期记忆、用户级偏好记忆、会话检索。

### 22.4 后续性能优化计划

P1：增加耗时打点。

```text
memory_load_ms
retrieval_ms
rerank_ms
parent_lookup_ms
llm_generate_ms
redis_write_ms
total_ms
```

P1：减少重复初始化。

- parent store 连接在应用启动时创建并复用。
- chat engine 中可复用的 retriever / reranker / postprocessor 不在每次请求重复构造。
- BM25 retriever 在服务启动时构建或缓存。

P1：优化检索参数。

- 拆分 `dense_top_k`、`sparse_top_k`、`fusion_top_k`、`rerank_top_n`、`final_top_k`。
- 减少进入 rerank 和 LLM 的无关上下文。

P2：缓存与流式体验。

- Redis 缓存高频查询结果。
- parent section lookup 批量查询或缓存。
- 优先优化 `/chat/stream` 的 TTFT。

### 22.5 面试表达

> 我在上线 MVP 阶段发现多轮问答响应偏慢，所以没有直接猜瓶颈，而是计划把一次请求拆成 memory、retrieval、rerank、parent lookup、LLM generate、Redis write 几个阶段分别打点。这样可以判断慢在检索、重排、父块回查还是大模型生成。后续根据瓶颈分别优化：BM25 持久化、检索参数收敛、parent lookup 缓存、流式输出和 prompt 压缩。
