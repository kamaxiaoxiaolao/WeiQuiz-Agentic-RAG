### 已完成的学习点

### 学到的面试问题

- 在 RAG 系统中，文档扫描阶段除了拿到文件路径，还应该记录哪些信息？为什么？
- 为什么文档扫描阶段要记录 `file_type`？
- 在 RAG 系统里，为什么不建议直接用文件名作为 `doc_id`？
- 如果没有显式 `doc_id`，你会怎么生成？

### 面试回答要点

- 文档扫描阶段不应该只记录文件路径，还应该记录 `file_name`、`file_type`、`file_size`、`mtime`、`doc_id` 等文档级 metadata。

- `doc_id` 是文档、chunk、索引记录和来源追踪之间的主键。
- 直接使用文件名作为 `doc_id` 容易在同名文件场景下冲突，建议优先使用显式稳定 ID，否则基于相对路径生成规范化 ID。

### 当前阶段

### 当前小节

Block -> Document -> Chunk metadata 标准化

### 已完成的学习点

- 理解了 LlamaIndex `Document` 不一定等于完整原始文件，它可以是一整个文件，也可以是一页、一段或一个自定义 `Block`。
- 理解了 LlamaIndex `TextNode` 基本可以理解为切分后的 chunk。
- 理解了当前项目的链路：`Block -> LlamaIndex Document -> TextNode(chunk) -> Milvus`。
- 理解了 `block_id` 和 `chunk_id` 的区别：
  - `block_id` 是解析层结构单元 ID。
  - `chunk_id` / `node_id` 是切分层 ID。
- 理解了为什么一个 chunk 未来可能对应多个 block，因此保留 `block_ids` 列表作为扩展点。
- 观察到 LlamaIndex 的 metadata 长度会影响有效 chunk size。测试中 `chunk_size=50` 时因为 metadata 太长触发异常，改为 `chunk_size=512` 后通过。

### 已完成的代码优化

- 在 `blocks_to_llama_documents()` 中为每个 `Block` 生成稳定 `block_id`。
- 在 Document metadata 中新增：
  - `block_id`
  - `block_ids`
- 当前 `block_ids` 只包含一个元素，后续可扩展为多个来源 block。
- 后续 chunk 会继承这些 metadata，形成：

```text
doc_id -> block_id/block_ids -> chunk_id
```

### 验证方式

  - 构造多个 `Block`

  - 验证 Document metadata 包含 `block_id` 和 `block_ids`
  - 调用 `chunk_documents()`
  - 验证 chunk metadata 继承 `block_id` 和 `block_ids`

### 学到的面试问题

- LlamaIndex 的 `Document` 是不是一定代表一个完整文件？
- LlamaIndex 里的 `Node` 和 chunk 是什么关系？
- 已经有 `chunk_id`，为什么还需要 `block_id`？
- 一个 chunk 如果对应多个 block，metadata 应该怎么设计？
- metadata 长度会不会影响 chunk size？

### 面试回答要点

- LlamaIndex `Document` 更准确地说是待索引文本单元，不一定是完整文件。
- `TextNode` 通常就是切分后的 chunk。
- 当前项目在 LlamaIndex 前有自定义解析层，PDF 会先解析成 title、text、table 等 `Block`。
- `block_id` 用于保留解析层结构单元，`chunk_id` 用于标识切分后的片段，两者粒度不同。
- 对企业级 RAG 来说，保留结构层级 ID 有助于溯源、排查解析质量、支持 parent-child chunk 和 Graph RAG 扩展。
- 当前实现中一个 Block 转一个 Document，所以一个 chunk 通常继承一个 `block_id`；但为了未来跨 block 合并，metadata 中同时保留 `block_ids` 列表。

### 当前小节

多格式文档解析与 parser 接入

### 已完成的学习点

- 明确了“能读取文件”和“能做结构化解析”不是一回事。
- 不同文档类型需要不同解析策略：
  - PDF：优先使用 `unstructured.partition.pdf` 提取标题、正文、列表、表格等结构；失败时 fallback 到 `pypdf`，保证至少能提取文本。
  - DOCX：使用标准库 `zipfile + xml.etree.ElementTree` 读取 `word/document.xml`，按 Word 段落样式和表格 XML 转换成结构化 Block。
  - HTML/HTM：使用标准库 `HTMLParser`，过滤 `script/style/nav/header/footer/noscript` 等噪声，提取标题、正文、列表、表格。
  - TXT/Markdown：保留简单文本读取路径，避免对结构化 Markdown 做过度清洗。
- 明确了 parser 的目标不是直接提升生成效果，而是降低数据污染，让后续 chunk、metadata、检索和排查更可靠。

### 已完成的代码优化

- 在 `app/ingest/document_parser.py` 中新增 DOCX parser：
  - 支持 `title`
  - 支持 `text`
  - 支持 `list_item`
  - 支持 `table`

### 学到的面试问题

#### 问题 1

你的 RAG 项目里，chunk size 和 chunk overlap 是怎么确定的？为什么不是随便设一个 512 或 1024？

#### 回答要点

- chunk size 是语义完整性、召回精度和成本之间的平衡。
- chunk 太小会导致标题、定义、表格、知识点被切散，检索到的上下文不完整。
- chunk 太大会混入无关内容，造成上下文污染，rerank 和 LLM 成本也会上升。
- overlap 用来缓解切分边界问题，让跨边界的句子或知识点有机会保留完整语义。
- overlap 过大会导致重复 chunk 增多、向量库膨胀、召回重复和成本上升。
- 参数选择不能靠拍脑袋，要通过 chunk_stats、Recall@K、MRR、引用准确率和检索延迟做对比。

#### 问题 2

为什么不能把解析后的每个 Block 直接作为一个 Document 进入 splitter？

#### 回答要点

- PDF parser 输出的 Block 可能非常碎，一个 Block 可能只是一行、页码、公式片段或短标题。
- 如果每个 Block 单独进入 splitter，后续 splitter 无法跨 Block 合并语义，会产生大量短 chunk。
- 当前项目通过 chunk_stats 发现平均 chunk 只有 22 个字符，短 chunk 占比很高，所以改为先聚合成 section-level Document，再切分。
- 这样可以保留标题、段落、表格之间的上下文关系，降低向量库污染。

#### 问题 3

metadata 是不是越多越好？为什么要精简入库 metadata？

#### 回答要点

- metadata 不是越多越好，入库 metadata 应该只保留检索过滤、展示和基础溯源必需字段。
- 过长 metadata 会增加向量库 payload，也可能影响 splitter 的长度预算。
- 当前项目保留轻量字段，例如 `doc_id`、`source_path`、`section_id`、`page_range`、`block_start_index`、`block_end_index`。
- 详细 `block_ids`、原始解析细节、调试信息更适合放在 audit/report，而不是全部塞进向量库。

### 下一步

- 将默认 chunk 参数从当前 `256/20` 调整为更合理的候选 baseline，例如 `512/50`，重新生成 chunk_stats 做对比。
- 后续再进入 parent-child chunk 设计，而不是马上三层 chunk 全量入库。

## 2026-05-07 Phase 2：检索评估闭环

### 当前小节

### 已完成的学习点
