## 2. 核心差距分析

### 2.1 Agentic 能力深度不足

**当前状态**: 线性 workflow，单一重试循环 (`Router → Retrieval → Quality → Rewrite(1次) → Generation`)




**企业级应有能力**:

#### 2.1.1 多跳推理 (Multi-hop Reasoning)

当前系统识别了 `multi_step` 意图，但没有实际的子问题分解逻辑。对于 "对比 A 方案和 B 方案的优缺点" 这类问题，企业级系统应：

- 将复杂问题拆解为多个子问题
- 分别对每个子问题执行检索
- 汇总多个检索结果进行综合推理
- 支持动态调整检索策略

#### 2.1.2 Self-Reflection / Self-RAG

当前只在检索前做质量检查，没有生成后的反思机制。企业级系统应：

- 生成回答后自检 "这个回答是否有源文档依据"
- 判断 "是否需要补充检索"
- 形成 `Generate → Reflect → Retrieve → Refine` 闭环
- 自动识别并修正幻觉内容

#### 2.1.3 规划能力 (Planning)

复杂任务需要动态规划和执行路径调整：

- ReAct 模式：Reasoning + Acting 交替执行
- Plan-and-Execute：先制定计划，再逐步执行
- 动态调整：根据中间结果修改后续计划

#### 2.1.4 多 Agent 协作

企业级系统通常采用多 Agent 架构：

- 不同 Agent 负责不同领域或工具
- 协调器 (Orchestrator) 统一调度
- Agent 间可传递上下文和中间结果

### 2.2 工具调用框架缺失

**当前状态**: Router 只做意图分类，`WEB_SEARCH` 和 `SQL_QUERY` 路由直接返回固定文案 "当前系统还没有接入..."

**企业级应有能力**:

| 能力 | 说明 |
|------|------|
| **Tool Calling 框架** | LLM 通过 function calling 动态选择和调用工具 |
| **可扩展工具注册表** | Web 搜索、SQL 查询、计算器、API 调用等工具可热插拔 |
| **工具结果注入** | 工具返回结果注入上下文后进行二次推理 |
| **工具调用链** | 支持多工具串联调用，前一个工具的输出作为后一个的输入 |
| **错误处理** | 工具调用失败时的降级策略和用户提示 |
| **可观测性** | 工具调用的完整 trace 记录 |

**示例场景**:
```
用户: "最近一个月的销售额是多少？"
系统:
  1. Router → SQL_QUERY
  2. 调用 SQL 工具查询数据库
  3. 获取结果后注入上下文
  4. LLM 生成自然语言回答
```

### 2.3 检索策略单一

**当前状态**: Dense (Milvus) + Sparse (BM25) 双路融合，固定 top_k=5

**企业级应有能力**:

| 策略 | 说明 | 当前状态 |
|------|------|----------|
| **多粒度检索** | 句子级、段落级、文档级多层索引 | ❌ 未实现 |
| **Multi-Vector Retrieval** | ColBERT、ColPali 等 token 级交互模型 | ❌ 未实现 |
| **Learned Sparse Retrieval** | SPLADE 等学习型稀疏表示 | ❌ 未实现 |
| **Contextual Retrieval** | 给每个 chunk 添加上下文前缀 (Anthropic 方案) | ❌ 未实现 |
| **动态 top_k** | 根据查询复杂度和置信度自适应调整 | ❌ 未实现 |
| **知识图谱增强 (GraphRAG)** | 实体关系图谱辅助检索 | ❌ 未实现 |
| **元数据过滤** | LLM 自动生成结构化过滤条件 (Self-Query) | ❌ 未实现 |
| **多索引路由** | 不同知识库使用不同索引，查询时动态选择 | ❌ 未实现 |

### 2.4 幻觉检测与 Grounding 不足

**当前状态**: 系统提示要求 "如果依据不足要说明"，但没有自动化检测机制

**企业级应有能力**:

#### 2.4.1 自动 Grounding 检查
- 将生成的每句话与源文档逐条核验
- 标记每句话的 grounding 状态 (fully grounded / partially grounded / not grounded)

#### 2.4.2 幻觉分类器
- 独立模型判断回答是否 hallucinated
- 区分事实性幻觉和推理性幻觉

#### 2.4.3 引用精确性验证
- 确保 `[来源 X]` 标注指向实际存在的相关内容
- 检查引用内容是否真的支持对应论点

#### 2.4.4 Faithfulness Score
- 量化回答对源文档的忠实度
- 低于阈值时自动触发重新生成

### 2.5 Guardrails 安全防护缺失

**当前状态**: 无内容安全机制

**企业级应有能力**:

| Guard 类型 | 功能 | 重要性 |
|------------|------|--------|
| **输入 Guard** | 提示注入检测、敏感信息过滤、恶意查询拦截 | P0 |
| **输出 Guard** | PII 脱敏、有害内容过滤、合规性检查 | P0 |
| **Topic Guard** | 限制回答范围只在企业知识库领域内 | P1 |
| **Rate Limiting** | 按用户的查询频率和并发数限制 | P1 |
| **Token Budget** | 控制单次查询的 token 消耗上限 | P2 |

| **用户画像** | 基于历史交互的个性化检索和回答 | ❌ 未实现 |

### 2.7 可观测性与监控不足

**当前状态**: WorkflowTrace 记录基本步骤信息，SSE 推送前端

**企业级应有能力**:

#### 2.7.1 专业可观测平台集成

| 平台 | 功能 |
|------|------|
| **LangSmith** | LLM 应用全链路追踪和评估 |
| **Arize Phoenix** | 检索和生成质量的可视化分析 |
| **Langfuse** | 开源 LLM 可观测性平台 |
| **Helicone** | LLM API 调用监控和成本分析 |

#### 2.7.2 全链路 Trace
- 从用户输入到最终输出的完整调用链
- 每步延迟、token 用量、成本统计
- 检索结果和生成内容的快照

#### 2.7.3 告警机制
- 检索质量下降告警
- 延迟飙升告警
- 错误率上升告警
- 成本异常告警

#### 2.7.4 A/B 测试框架
- 不同检索策略的效果对比
- 不同模型的效果对比
- 统计显著性检验

### 2.8 多租户与权限控制粗糙

**当前状态**: user/admin 两级 RBAC，文档上传需要 admin 权限

**企业级应有能力**:

| 能力 | 说明 |
|------|------|
| **文档级 ACL** | 不同用户/角色只能检索到权限范围内的文档片段 |
| **Chunk 级权限** | 继承自源文档的细粒度访问控制 |
| **多租户数据隔离** | 不同部门/组织的知识库完全隔离 |
| **审计日志** | 谁在什么时候查了什么、看到了什么 |
| **权限继承** | 文档夹/标签级别的权限继承机制 |

### 2.9 评估体系不完整

**当前状态**: 基础的 Hit@K / MRR / Recall 评估，使用外部数据集 (MSMARCO/SQuAD)

**企业级应有能力**:

#### 2.9.1 端到端 RAG 评估 (RAGAS 框架)

| 指标 | 说明 |
|------|------|
| **Faithfulness** | 回答是否忠实于检索到的上下文 |
| **Answer Relevance** | 回答是否与问题相关 |
| **Context Recall** | 检索的上下文是否覆盖了正确答案 |
| **Context Precision** | 检索的上下文中相关内容的比例 |

#### 2.9.2 业务数据评估
- 基于企业自身 QA 对的评估，而非通用数据集
- 覆盖企业典型查询场景
- 定期更新评估集

#### 2.9.3 回归测试
- 每次索引更新后自动运行评估
- 每次模型切换后自动运行评估
- 质量指标下降时自动告警

#### 2.9.4 人工标注闭环
- 用户反馈 → 标注 → 微调评估集
- 持续改进评估覆盖度

### 2.12 缺少的高级 Agentic 模式

| 模式 | 说明 | 当前状态 |
|------|------|----------|
| **Corrective RAG (CRAG)** | 检索结果不佳时自动切换到 Web 搜索或其他知识源 | ❌ 未实现 |
| **Adaptive RAG** | 根据查询特征动态选择不同的检索和生成策略 | ❌ 未实现 |
| **Speculative RAG** | 多个 Generator 并行生成，选出最优答案 | ❌ 未实现 |
| **Self-Query** | LLM 自动生成结构化元数据过滤条件 | ❌ 未实现 |
| **Sub-question Decomposition** | 将复杂问题自动分解为多个子问题分别检索 | ❌ 未实现 |
| **Hypothetical Document Embedding (HyDE)** | 先生成假设性答案，再用其进行检索 | ❌ 未实现 |

## 4. 改进路线图建议

### 第一阶段：安全与可信 (4-6 周)

**目标**: 确保系统输出安全可信，满足企业基本合规要求

1. **Guardrails 框架**
   - 集成 NeMo Guardrails 或 Guardrails AI
   - 实现输入提示注入检测
   - 实现输出 PII 脱敏和有害内容过滤
   - 添加 Topic Guard 限制回答范围

2. **Grounding 检查**
   - 实现回答与源文档的自动核验
   - 添加 Faithfulness Score 计算
   - 低分回答自动标注警告

3. **文档级权限**
   - 在检索时注入用户权限过滤条件
   - 实现 Chunk 级权限继承
   - 添加审计日志记录

### 第二阶段：智能增强 (6-8 周)

**目标**: 提升复杂问题的处理能力

1. **工具调用框架**
   - 实现 Function Calling 工具注册机制
   - 集成 Web 搜索工具 (Tavily / SerpAPI)
   - 集成 SQL 查询工具

## 5. 总结

### 5.1 项目优势

本项目已经搭建了一个**结构清晰的 Agentic RAG 骨架**，以下方面达到了中等水平：

- ✅ 混合检索 (向量 + BM25 + 融合 + 重排)
- ✅ 层级分块与 Auto-Merging
- ✅ Query Router 与 Query Rewrite
- ✅ LlamaIndex Workflow 编排
- ✅ SSE 流式步骤推送
- ✅ JWT 认证与 RBAC
- ✅ Docker Compose 全栈部署

### 5.2 主要差距

距离企业级生产环境主要差距在于四个维度：

| 维度 | 差距描述 |
|------|----------|
| **Agentic 深度** | 只有单一重试循环，缺少多跳推理、自我反思、动态规划 |
| **安全合规** | 无 Guardrails、无幻觉检测、权限控制粗糙 |
| **生产就绪度** | 缺少语义缓存、限流、结构化日志、可观测平台 |
| **评估闭环** | 无法量化 "系统变好了还是变差了"，缺少持续改进机制 |

### 5.3 建议

建议按 **P0 → P1 → P2 → P3** 的顺序逐步补齐：

1. **优先确保安全和可信度** (Guardrails + Grounding + 权限)
2. **再提升智能程度** (工具调用 + 多跳推理 + 评估)
3. **然后优化工程化** (可观测 + 缓存 + 并发)
4. **最后实现深度智能** (记忆 + 图谱 + 自进化)
